# SPEC — 회의 조율 제품 (토스 프로덕트 디자이너 과제)

> **이 문서가 정본이다.** 대화 기록·구현 코드·디자인 파일과 충돌하면 이 문서가 우선한다.
> 규칙을 바꾸고 싶다면 구현을 먼저 바꾸지 말고 이 문서를 먼저 수정한다(관문 원칙).

## 0. 과제 컨텍스트

- 과제: 같은 회사 동료 6명이 다음 주까지 1시간 회의를 잡는 경험 설계.
  단서: 점심 직후 회피(선호) / 특정 요일 외근(제약) / 필수 참석 vs 선택 참석 / "동료들의 상황을 파악하고 → 시간을 찾아 잡는" 2단계.
- 평가 기준: 문제 정의, 솔루션 설계(명확한 가설과 의도), 시각적 완성도.
- 제출: 사용 흐름이 보이는 URL + 질문 3개(발견한 문제와 이유 / 어떻게 해결 / 왜 이 방식).
- 토스 공식 팁: "쓰기 쉬운 일정 관리 제품을 만든다고 가정. 단편적 문제 해결보다 근본적으로 문제가 발생하지 않는 제품 구조."

## 1. 문제 정의

원문 단서 4개 분석:
1. **제약의 이질성** — "점심 직후 회피"(선호)와 "외근"(물리적 제약)은 확실성의 정도가 다르다.
2. **참석자의 이질성** — 필수/선택 구분은 "안 맞으면 누굴 빼는가"라는 우선순위 판단을 요구한다.
3. **정보의 소재지** — 조건들은 주최자 머릿속이 아니라 동료 각자에게 흩어져 있다.
4. **프로세스의 단계성** — 파악 → 조율/확정. 정보 수집 자체가 독립된 어려움이다.

보강 2개:
5. **탐색 공간의 유한성** — "다음 주 + 1시간 + 근무시간" = 후보 약 40슬롯. 6명의 제약이 각각 슬롯을 소거하면 전원 만족 교집합이 공집합일 확률이 높다. 따라서 중심 문제는 "빈 시간 찾기"가 아니라 "없을 때 무엇을 양보할지 결정하기"다.
6. **커뮤니케이션 비용은 제거되지 않고 이전된다** — 주최자의 취합 비용을 참여자 6명의 입력 비용으로 분산하면, 응답 동기와 미응답 처리라는 새 비용이 생긴다. 미응답은 두 병목(취합·판단)이 교차하는 지점(불완전한 정보로 판단해야 하는 상황)이다.

기존 캘린더 도구가 못 푸는 이유: (a) 암묵적 선호 등 "말로 표현되지만 기록되지 않은 정보"를 확인·처리하지 못하고, (b) 이질적 조건 간 우선순위를 판단해 트레이드오프를 결정해주지 않는다. 그리고 (c) 기록되지 않는 근본 원인은 기능 부족만이 아니라 **공개 구조** — 모두에게 보이는 캘린더에 적기 애매한 정보이기 때문이다(→ R4 공개 범위의 근거).

## 2. 가설

**문제 가설.** 주최자는 6명 각자에게 흩어진, 서로 다른 성격(선호/제약)과 무게(필수/선택)를 가진 조건들을 일일이 물어보고 머릿속에서 종합해 "모두가 괜찮은" 시간을 판단해야 한다. 주어진 시간 조건상 모두를 만족하는 시간이 존재하지 않을 가능성이 높기 때문에, 이 판단은 빈 시간 탐색이 아니라 "무엇을 포기하고 무엇을 지킬지"의 트레이드오프 결정이 되고, 이 취합→판단→조율 과정은 번거롭고 부담스럽고 실수하기 쉽다.

**설계 가설.** 조건을 각 참여자에게 말하듯 가볍게 입력받아 시스템이 성격(확실성: 불가/회피)과 무게(중요도: 필수/선택)로 분류하고, 전원 만족 시간이 없을 때 "이 안은 무엇을 포기하는지"가 명시된 소수의 후보로 계산해 제시하면 — 주최자는 취합·독촉·머릿속 종합 없이 후보 중 선택만으로 회의를 확정할 수 있다.

하위 가설 3개:
- **입력 가설** — 자연어로 받고 분류는 시스템이 하되 결과를 칩으로 확인·수정하게 하면, 참여자는 분류 인지 부담 없이 30초 내로 응답을 마친다.
- **판단 가설** — 후보를 포기 항목 명시형("전원 참석이지만 A의 선호 양보" vs "선호 전부 충족이지만 선택 참석자 B 제외")으로 제시하면, 주최자는 조건 재검토 없이 후보 비교만으로 결정한다.
- **경계 가설** — 계산은 시스템이 전담하되 확정(특히 사람을 빼는 결정)은 주최자에게 남기면, 자동화의 효율과 사회적 결정의 책임 소재가 양립한다.

핵심 프레임: 좋은 트레이드오프 화면은 "어려운 선택"이 아니라 "성격이 다른 선택"을 보여준다. 선택이 쉬워 보이는 것 자체가 시스템의 성과다(도구 없이는 그 명백함에 도달하는 것부터가 일이었으므로). 비유: 내비게이션 — 경로 계산은 시스템, 최종 선택은 운전자.

## 3. 확정 규칙 (R1–R8)

### R1. 양보 비용 서열
- **규칙**: 양보 불가 = 물리적 제약(불가), 필수 참석자의 참석. 양보 비용 오름차순 = 선택 참석자의 선호 < 필수 참석자의 선호 < 선택 참석자의 참석. 비용이 낮은 것부터 양보한다.
- **후보 생성(최소 제외 집합)**: 후보 슬롯 = 필수 참석자 전원이 하드 프리인 슬롯. 각 후보의 제외 집합 = 그 슬롯을 하드 차단하는 선택 참석자들(계산으로 유일하게 결정 — 부분집합 탐색 불필요). 비용 = 제외 인원 × 선택 참석 가중치 + 남는 참석자의 선호 위반(사람 단위) 가중 합. 전원 참석(제외 0명)과 다중 제외는 같은 공식의 특수/일반 케이스다. 제외된 사람의 선호 위반은 계산하지 않는다(참석하지 않는 사람의 선호 위반은 성립하지 않음).
- **가중치 불변식**: 선택 참석 가중치 > 선택 선호 가중치. 선호 회피 목적의 자발적 제외가 절대 이득이 되지 않도록 보장한다.
- **비용 집계 단위**: 랭킹·비교 집계는 **사람 단위**(슬롯당 1인 1회 — 같은 슬롯을 덮는 칩이 여러 개여도 비용은 1인분). 조건 건수(칩 단위)는 상세 표기에만 쓴다. 칩을 남발해도 비용이 늘지 않으므로 R4의 악용 대응과 정합적이다.
- **동순위 정렬**: ① 양보 비용 합계 오름차순 → ② 동률이면 이른 날짜·시간 순.
- **근거**: 과제 원문 "모두가 괜찮은 시간" = 참석 포괄성이 1차 목표. 선호 양보는 회복 가능(사과 한마디), 사람 제외는 통보되는 사회적 행위라 비가역성이 크다. 다중 제외를 허용하는 이유: 선택 참석자의 참석은 R1이 금지하는 항목이 아니라 비용이 큰 양보 항목이며, 서열상 뒤로 밀려 접힌 대안으로 노출될 뿐 은폐할 이유가 없다(R2의 "숨기지 말고 명시" 원칙).
- **기각 대안**: "필수 참석자 선호 > 선택 참석자 참석" 보호 서열(초기 예시) — 검산 결과 추천이 뒤집히는(사람 제외안 추천) 것을 확인하고 기각. / 주최자의 "n명 이상 제외 금지" 필터 — 같은 의미의 통제 장치가 이미 존재(빠지면 안 되는 사람 = 필수 지정, R3의 참석 가중치 정의 그 자체)하고, 후보를 계산 전에 잘라내는 방식은 "숨기지 말고 명시 + 선택은 주최자"라는 R2와 결이 다르며, 필수/선택 지정과 필터가 충돌하는 상태를 새로 정의해야 하는 문제만 만든다.

### R2. 자동화의 경계
- **규칙**: 계산·순위화·후보 압축은 전자동. 확정(특히 사람을 빼는 결정)은 주최자가 한다. 추천 후보군 1개 + 원탭 확정 + 접힌 대안 후보군 최대 2개 구조.
- **후보군 그룹핑(표시 레이어 규칙 — 엔진은 슬롯 단위로 전부 계산)**: 그룹 키 = (제외되는 참석자 집합, 선호가 반영되지 않는 사람 집합). 키가 같으면 트레이드오프 관점에서 정보량이 동일한 슬롯이므로 카드 하나로 묶는다. 그룹 키가 같으면 R1 비용은 자동으로 동일하다(불변식 — 엔진에서 assert). 조정가능 메타(R3)는 그룹 키에 포함하지 않는다 — 조정가능이 붙은 하드 제약 슬롯은 애초에 후보가 아니며(계산상 불가), 조정가능은 후보군 속성이 아니라 탈출구의 재료다.
- **그룹 랭킹** = (그룹 비용, 그룹 내 최이른 슬롯) 오름차순 — R1 동률 규칙의 확장. 그룹 내 기본 선택 = 최이른 슬롯, 그룹 내 다른 시각은 카드 안에서 선택. 원탭 확정은 현재 선택된 시각을 확정한다.
- **대안 선정** = 랭킹 순 다음 후보군 최대 2개. 포기 "유형"이 다른 것만 노출하는 규칙은 두지 않는다 — 그룹 키가 다르면 포기 내용이 이미 다르고, 유형 강제는 더 순위가 높은 대안을 숨길 수 있다. (시드의 "상위 두 후보의 포기 종류가 다름"은 시드 저작 스펙이지 엔진 규칙이 아니다.)
- **노출 개수**: 그룹당 시간 3개 노출 + 초과분은 "n개 더 보기". 단, 숨는 것이 1개뿐이면 전부 노출한다("1개 더 보기"는 어색함).
- **한 줄 모드의 통합**: 완벽 슬롯이 존재하면 그것은 "포기 0건 후보군" 하나다. "모두 괜찮은 시간이 있어요. [요일] [시간], 확정할까요? (다른 시간 n개)" 한 줄 모드는 그룹핑의 특수 케이스로 동작한다.
- **빈 상태**: 필수 참석자끼리의 제약이 겹쳐 후보 슬롯이 0개인 경우(자유 모드에서 발생 가능)에만 "가능한 시간이 없어요 — 필수/선택 분류나 조건을 조정해보세요" 안내를 표시한다. 최소 제외 집합 규칙 덕에 선택 참석자 제약만으로는 공집합이 되지 않는다.
- **기각 대안(그룹핑)**: 같은 포기의 슬롯을 카드로 전부 펼치기 — 후보 압축에 역행. / 그룹 대표 시각 하나만 노출 — 유효한 선택지를 숨김.
- **근거**: 사람 제외는 시스템이 책임질 수 없는 사회적 행위("앱이 널 뺐어"는 조직에서 성립하지 않음). 규칙이 화면에 드러나야 함(포기 항목 명시가 규칙의 시각화). 시스템이 모르는 맥락(이번 회의에 그 사람이 실제로 필요한가)은 원래부터 주최자만 답할 수 있는 질문이다.
- **기각 대안**: 완전 자동 확정 — 규칙이 보이지 않게 되고 사회적 책임 소재가 무너짐.

### R3. 조건 모델과 확실성 판별
- **규칙**: 조건 = 시간 × 확실성{불가/회피} × 조정가능 메타 × 출처{캘린더/응답}. 사람 = 참석 가중치{필수/선택}(주최자가 회의 생성 시 지정).
- 확실성은 일정의 종류가 아니라 **화자의 표현**으로 판별: "안 돼요"=불가, "피하고 싶어요"=회피, 캘린더 일정=기본 불가.
- 조정 가능 = 하드 제약에 붙는 메타데이터("불가인데 움직일 수는 있음"). **선택적 보너스 정보** — 강제하지 않는다. 아무도 안 말하면 전 일정 고정(현실과 동일한 기본값), 말한 것만 탐색 공간을 넓힌다.
- **기각 대안**: 폼 입력(내용 입력 + 시간 드롭다운 + 조정가능 체크박스) — "네 조건을 분류해 입력하라"는 인지 부담이 입력 가설(30초)을 죽임. 칩 목록이 이미 폼이며, 차이는 초안을 시스템이 채운다는 것뿐. / 조정가능 전수 확인 — 일정 수만큼 질문이 늘고 응답률 하락으로 잃는 정보가 더 많음.

### R4. 공개 범위 (3단계 가시성)
- **규칙**: 본인 = 전부(자기 캘린더·원문·칩). 주최자 = 시스템이 정제한 조건만("서연: 13시 회피") — 사유·원문 비노출, 타인 캘린더는 가능/불가만(일정 제목 비노출). 다른 참여자 = 개인 단위 정보 없음, 집계(응답 현황)와 확정 결과만.
- **근거**: 공개된다고 느끼면 회피 조건을 안 적는다 → 입력 데이터가 기존 캘린더와 같아져 제품 존재 이유 소멸. 이 제품이 빚진 투명성은 "결정이 어떻게 내려졌는가"(규칙·결과)이지 "각자의 사정"이 아님. 조건 상호 공개는 투명이 아니라 사회적 압력을 만든다. 비공개 기본값 + 자발적 공개 = 기존 조직 규범과 일치.
- **악용(회피 남발) 대응**: R1상 회피는 거부권이 없음(충돌 시 가장 먼저 양보됨). 트레이드오프 화면에 "이 안은 ○○ 님 회피 n건과 충돌"처럼 조건 건수·이름이 가시화되어 사유 없이도 패턴은 드러난다.
- **동률·맥락 필요 시 출구**: 열람 확대가 아니라 "콕 집어 다시 묻기"(리마인드 채널로 특정 질문 발송, 본인이 답장에서 공개 범위를 스스로 결정).
- **기각 대안**: 주최자 한정 사유 공개 — 참여자 신뢰 문구와 모순, 주최자의 행동을 바꾸지 못하는 정보(회피를 존중/양보 중 택일인 것은 동일).

### R5. 캘린더 경계 (조율 레이어 정체성)
- **규칙**: 사내 캘린더는 별개 제품. 읽기 = 프리필(전 참여자 확정 일정). 쓰기 = **자기가 만든 객체만** — 확정된 회의 1건을 참석자 캘린더에 등록. 그 외 일체 쓰기 없음.
- 정정 칩([이 시간 비어 있어요]/[옮길 수 있어요])은 일정 편집이 아니라 **이 회의에 대한 답변** — 이 회의의 계산 레이어에만 반영, 원본 불변. 카드는 삭제되지 않고 취소선+실행 취소로 남는다(한 탭 복구 + 기록 vs 진술 구분 유지).
- [비어 있어요] 선택 직후 "사내 캘린더에서 열기" 딥링크 노출(쓰기 권한 없는 이동, 보조 링크, 강요·모달 아님, 투어 미포함).
- **근거**: ① "비어 있어요"의 원인(취소/이동/불참 가능)을 구분하지 않아도 계산에는 충분하며, 삭제 동기화하려면 구분 질문이 필요해 입력 가설이 깨짐. ② 파괴적 쓰기의 비대칭 — 낡은 일정 잔존 비용 << 오삭제 비용. ③ 삭제 권한은 기술적으로 존재하나(캘린더 API 쓰기 스코프) 실무 관행은 최소 권한이며(스케줄링 도구는 자기 생성 이벤트만 조작), 핵심 가치가 아닌 파괴적 권한은 받지 않는 것이 신뢰 비용상 이득.
- **기각 대안**: 양방향 상시 동기화 — 회피는 애초에 캘린더에 쓸 수 없는 정보(R4)이고, 동기화가 오히려 확인·충돌·권한이라는 진짜 이중 관리 문제를 만든다.
- 구현: 실제 API 연동 금지. 시드 데이터의 source 필드로 시뮬레이션, 확정 시 "참석자 캘린더에 등록됐어요" 라벨로 컨셉 전달.

### R6. 출처 구분 (기록 vs 진술)
- **규칙**: 캘린더 출처와 응답 출처를 데이터·UI 모두에서 구분한다. 구분은 기호가 아니라 **행동과 문구**로 드러낸다: 진술 칩은 편집·삭제 가능, 캘린더 카드는 답변(정정)만 가능. 붕괴 장면의 설명은 "캘린더에 없던 일정이 발견됐어요".
- **근거**: 이 구분이 없으면 확정 기능 3개가 불능 — 검수·정정(뭐가 내 말인지 보여야 고침), 행동 일관성(탭 결과 예측 가능), 미응답 디폴트 규칙(출처 필드 필요).

### R7. 미응답 처리
- **규칙**: 예방 = 입력 비용 최소화(자연어 한 줄, 로그인 없음, 30초, 응답 기한 표시). 치료 = 리마인드(주최자 수동 + 데드라인 임박 자동) + 디폴트 규칙(미응답자는 캘린더 확정 일정만 반영, 선호는 없는 것으로 간주해 계산, "잠정" 배지 표시).
- **근거**: 미응답 원인은 "귀찮음"(설계로 제거)과 "잊음/바쁨"(장치로 대응)의 두 종류라 둘 다 필요하다.

### R8. 열린 문서 모델과 확정·재조율
- **규칙**: 확정 전 — 참여자 응답은 열린 문서(같은 링크로 재진입해 조건 추가·수정, 실시간 재계산). "일정 등록" 개념 없음 — 새 일정은 새 [불가] 조건일 뿐.
- 확정 = 조율 세션의 종료이지 데이터의 폐기가 아님: 회의가 캘린더에 등록되고, 응답 문서는 읽기 전용이 되며, 조건 데이터는 보존된다.
- 재조율 열쇠는 주최자의 **"다시 조율하기"** 단일 진입점: (a) 주최자 쪽 변수 → 직접 해제, 보존된 조건 위에서 재계산 + "조건이 바뀐 분만 알려주세요" 재확인 알림. (b) 참여자 쪽 변수 → 잠긴 응답 화면의 "확정된 시간에 참석이 어려워졌어요" 신고 버튼 → 주최자에게 알림 → 재조율 여부는 주최자가 판단(경계 가설의 적용: 회의를 다시 흔드는 것은 사회적 결정).
- 재조율 비용 = 전체 재수집이 아니라 재계산 한 번(조건 보존 구조의 부산물, "근본적으로 문제가 발생하지 않는 구조"의 사례).
- 구현 범위: 재조율 플로우 화면은 만들지 않는다. UI 흔적 최대치 = 참여자 완료 화면의 신고 버튼 한 줄 + 주최자 확정 화면의 "다시 조율하기" 진입점 한 줄 + "변수가 생기면 이 조건들로 다시 계산해드려요" 안내 문구.

## 4. 화면 브리프 (4종)

> 아래는 브리프(반드시 담을 요소와 연결 규칙)다. 시각 설계는 클로드 디자인에서 검증하며 확정하되, 요소·규칙의 삭제는 SPEC 수정을 선행할 것.

### 4.1 참여자 입력 (모바일 기준) — 가장 상세히 논의됨
5층 구조: 맥락 → 확인 → 진술 → 검수 → 제출.
1. **맥락 헤더**: 요청자, 회의명, 길이, 범위("다음 주 중"), 응답 기한(동기 장치).
2. **프리필**: 캘린더 출처 일정을 일자별 세로 리스트로, 일정 있는 날만 카드·없는 날은 한 줄로 접기("화–금 일정 없음"). 카드 라벨 "캘린더 · 탭해서 정정" = 출처 표시 + 정정 진입점. 탭 → [비어 있어요]/[옮길 수 있어요] → 취소선+실행 취소, [비어 있어요] 직후 딥링크(R5).
3. **자연어 입력**: 질문 문구 "여기 없는 일정이나 피하고 싶은 시간이 있나요?"(차분만 말하게 유도). 빈 제출 허용(캘린더만으로 응답 완료).
4. **칩 검수**: "이렇게 이해했어요 — 탭해서 고칠 수 있어요". 칩 = 불가(적색계)/회피(황색계)/병합(무채색, "캘린더에 이미 있어요"). 칩 옆에 원문 단서 흐리게 표시(추적성). 탭 → 성격 전환·시간 수정·삭제.
5. **신뢰 문구 + 제출**: "주최자에게는 조건만 전달돼요. 이유와 원문은 보이지 않아요."를 제출 버튼 바로 위에(망설임이 생기는 위치에서 답한다). 제출 후 "확정 전까지 언제든 수정할 수 있어요".
상태 3종: 분류 실패(회색 미분류 칩으로 남기고 시간 지정 요청 — 조용히 버리지 않음), 제출 완료(요약 + 수정 진입점 + 확정 후에는 신고 버튼), 정정 후(취소선 카드 + 딥링크).

### 4.2 주최자 대시보드 (데스크톱 기준)
필수 요소: 취합 현황(6명 중 n명, 개인별 응답 여부), 수집된 조건의 가시화(누구의 어떤 조건 — R4 범위 내: 조건만, 사유 없음), 미응답자 표시 + [리마인드 보내기](투어 전환 트리거), 잠정 추천 배지("확정 일정 기준 수 14시 — 도윤 님 응답 전이에요", R7), 데드라인, 회의 생성 시 필수/선택 지정 이력.

### 4.3 트레이드오프 후보
필수 요소: **후보군 카드**(R2 그룹핑 — 추천 후보군 1개 강조 + 접힌 대안 후보군 최대 2개, 그룹 내 시간 선택·기본 선택 최이른 슬롯·"n개 더 보기") + 원탭 확정.
카드 정보는 **2층 구조**:
- 1층(비교용 집계, 사람 단위, 분모 고정): "참석 6/6" / "참석 5/6 · 도윤 님(선택) 제외" / "참석 4/6 · 도윤·수아 님(선택) 제외". 분모는 항상 전체 초대 인원 — 카드마다 분모가 흔들리면 비교 기능이 죽는다.
- 2층(구체 포기, 주체 명시): "서연 님 선호 1건 미반영". 조건 건수(칩 단위)는 이 층에서만 쓴다(R1 집계 단위 원칙). cost 숫자는 어떤 층에도 노출하지 않는다.
그 외: 각 후보의 포기 항목 명시(R2 — 규칙의 시각화), 충돌 조건의 주체·건수 표시(R4 악용 대응), 미응답자가 있으면 카드에 "잠정" 배지 승계(R7), 후보 카드에서 참석자 분류(필수/선택) 확인·정정 → 즉시 재계산, "콕 집어 다시 묻기" 진입점(자유 모드 깊이, 투어 미포함).
전원 만족 슬롯이 존재하는 입력에서는 "목요일 2시, 모두 괜찮아요. 확정할까요? (다른 시간 n개)" 한 줄 모드 = 포기 0건 후보군(R2). 후보 슬롯이 0개면 빈 상태 안내(R2).
**기각 대안(지표)**: "양보 없이 참석 n/m명" 독립 지표 — 분모가 카드마다 달라지고, 2층의 구체 포기 문구와 완전 중복이며, cost 비노출 원칙을 숫자로 우회 침식한다.

### 4.4 확정
필수 요소: 확정 결과 요약(시간, 참석자 명단 — 제외자가 있으면 드러남), "참석자 캘린더에 등록됐어요" 라벨(R5), 참여자 통보 문구는 절차 투명성 수준("전원 참석 가능한 시간 기준으로 정해졌어요"), "다시 조율하기" 진입점(R8), "변수가 생기면 이 조건들로 다시 계산해드려요" 안내 한 줄.

## 5. 데모 연출 스펙

- **형식**: 실제 동작하는 웹 프로토타입(피그마 아님 — 계산이 진짜여야 함). 주 제출물 = 동작 URL, 보조 = 1~2분 워크스루 영상(프로토타입 미작동 대비 보험). 기준 환경 데스크톱, 모바일 접속 시 축약 대응 또는 "PC 권장" 안내.
- **무대 구조**: 데스크톱 한 화면. 주최자 뷰 상주. [리마인드 보내기] 클릭 → 주최자 뷰가 뒤로 물러나 어두워지고 폰 프레임이 슬라이드 인(도윤의 모바일). 제출 → 폰 아웃, 주최자 뷰 복귀. 전환은 토글이 아니라 평가자 행동이 만든 인과.
- **투어 3비트** (가이드 오버레이, 특정 영역 하이라이트 + 나머지 잠금):
  1. 주최자 뷰에서 시작 — 5명 응답 완료·도윤 미응답·데드라인·잠정 추천 수 14시 배지. 맥락 먼저.
  2. [리마인드 보내기] → 폰 프레임 → 평가자가 도윤으로서 자연어 입력, 칩 3종(불가 1 + 병합 1 + 회피 1) 분류·병합 체험.
  3. 복귀 → "도윤 님의 응답이 도착했어요" → 재계산 → 잠정 추천 붕괴(원인 서술: 캘린더에 없던 일정 — 평가자가 방금 입력한 그 조건) → 트레이드오프 화면(금 13 추천 + 수 14 대안) → 원탭 확정 → "직접 사용해보세요" 잠금 해제.
- **오버레이 포함/제외**: 포함 = 위 3비트(미응답자 입력→재계산이 뼈대), 리마인드/디폴트 규칙은 ②에서 짧게 노출, 칩 검수는 ①에서 자연 체험. 제외(자유 모드 발견용) = 쉬운 케이스 전환·탈출구, 트레이드오프 화면의 분류 정정→재계산, 다시 묻기. 오버레이 마지막 카드에 "조건을 바꿔보세요 — 후보가 실시간으로 다시 계산됩니다" 한 줄(방향 지시 없이).
- **자유 모드**: 모든 인터랙션 해제, 조건 수정·미응답 대리 입력 → 실시간 재계산, 탈출구로 트리비얼 전환 목격 가능. 구석에 "처음부터 다시 보기"(시드 리셋 + 투어 재시작). 상태는 메모리에만(서버 없음, 새로고침 = 초기화).
- 원칙: 오버레이 = 문제 서사의 최단 경로. 그 외 전부 = 자유 모드에서 발견되는 깊이.

## 6. 시드 시나리오 (요약 — 정본 데이터는 seed.json, 검증은 verify_seed.py)

## 8. 제출 답변 3문항 재료

1. **어떤 문제, 왜 문제인가**: §1 (4단서 + 탐색 공간 유한성 계산 + 기록되지 않는 정보의 원인이 공개 구조라는 발견).
2. **어떻게 해결했나**: §2 설계 가설 + §3 규칙(분산 입력, 칩 분류, 포기 명시형 후보, 잠정 추천→붕괴→트레이드오프).
3. **왜 이 방식인가**: 판단 비용은 시스템이 흡수하되 되돌릴 수 없는 사회적 결정의 권한은 사용자에게 남김(R2) / 시스템이 모든 맥락을 아는 척하는 대신 아는 것과 모르는 것의 경계를 화면에 정직하게 드러냄 / 조건은 비공개로 들어가고 결정만 밖으로 나온다 — 시스템이 대인 비용의 완충재(R4) / 조건 보존 구조라 재조율이 재계산 한 번 — 근본적으로 문제가 발생하지 않는 구조(R8) / 캘린더의 대체재가 아니라 위에 얹히는 조율 레이어 — 읽어오고, 조율하고, 결과만 되돌려줌(R5).
