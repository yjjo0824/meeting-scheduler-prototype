# CLAUDE.md — 회의 조율 제품 (토스 PD 과제)

## 정본 원칙
- **SPEC.md가 정본이다.** 이 파일·대화 기록·코드와 충돌하면 SPEC.md가 우선한다.
- **변경의 관문**: 규칙·스코프를 바꾸려면 구현보다 SPEC.md 수정이 먼저다. SPEC의 각 규칙에는 근거와 기각된 대안이 있으니, 뒤집으려면 그 근거부터 반박할 것.
- **엔진 완료 조건**: `python3 verify_seed.py`가 ALL CHECKS PASSED여야 한다. 엔진·시드를 수정했다면 반드시 재실행.
- **커밋 규칙**: 기능 하나를 완성하고 동작을 확인할 때마다 git commit을 하고, 원격 저장소(GitHub)가 연결되어 있으면 push도 함께 한다. 커밋 메시지는 무엇을 했는지 한국어 한 줄로 쓴다.

## 절대 규칙 요약 (상세와 근거는 SPEC.md §3)
1. **R1 양보 비용**: 불가·필수 참석은 양보 불가. 양보 순서 = 선택 선호 → 필수 선호 → 선택 참석. 동률이면 이른 시간.
2. **R2 경계**: 계산은 전자동, 확정은 주최자. 후보에는 포기 항목을 반드시 명시.
3. **R4 공개 범위**: 사유·원문은 누구에게도 비노출. 주최자는 조건만, 참여자 상호는 집계·결과만.
4. **R5 캘린더**: 읽기 전용 + 자기가 만든 확정 회의만 쓰기. 정정 칩은 원본 불변(계산 레이어만). 실제 API 연동 금지(source 필드 시뮬레이션).
5. **R8 열린 문서**: 확정 전 응답은 수정 가능·실시간 재계산. 확정 = 잠금 + 데이터 보존. 재조율은 주최자의 "다시 조율하기"로만.

## 파일 맵
- `SPEC.md` — 문제 정의·가설·규칙 R1~R8·화면 브리프 4종·데모 연출·기각 목록·제출 답변 재료
- `seed.json` — 6명 페르소나, 회의 컨텍스트, 기대 계산 결과 (데모 시드의 유일한 출처 — 하드코딩 금지)
- `verify_seed.py` — 정답지 테스트 + R1 랭킹 참조 구현
- `IMPLEMENTATION_SPEC.md` — 화면·상태·구현 순서 보조 명세. SPEC.md와 충돌 시 SPEC.md 우선.

## 구현 시 주의
- 시드 수치를 코드에 하드코딩하지 말 것 — 항상 seed.json에서 로드. 자유 모드에서 조건이 바뀌면 엔진이 일반 규칙으로 재계산해야 한다(시드는 스트레스 테스트지 정답 분기가 아님).
- 데모 상태는 메모리에만 유지(localStorage 금지). "처음부터 다시 보기" = seed.json 리로드 + 투어 재시작.
- 투어 시작 상태: 5명 응답 완료, 도윤(doyun) 미응답(`responded_at_demo_start: false`), 잠정 추천 수 14시 배지.
- 전원 만족 슬롯이 존재하는 입력에서는 트레이드오프 화면이 아니라 "고민할 것 없음" 한 줄 모드로 전환할 것(SPEC §4.3).
